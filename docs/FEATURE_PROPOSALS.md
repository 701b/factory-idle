## FP-001: 자원 지대 탐색 시스템 (Resource Deposit Discovery)

### 배경

현재 설계에서는 채굴기를 연구 해금 후 무한정 건설할 수 있어 자원 추출에 물리적 제한이 없다.
이는 초중반 밸런싱을 어렵게 만들고, 플레이어에게 의미 있는 병목을 제공하지 못한다.

### 핵심 아이디어

- **자원 지대(Deposit)** 개념 도입: 채굴기는 자원 지대 위에만 건설 가능
- 자원 지대 1곳당 채굴기 **1대** 배치
- **탐색(Exploration)**을 통해 새로운 자원 지대를 발견

### 상세 설계

#### 자원 지대 (Resource Deposit)

| 속성 | 설명 |
|------|------|
| `depositId` | 고유 식별자 |
| `resourceType` | 채굴 가능 자원 종류 (iron, copper, coal) |
| `discovered` | 발견 여부 |

#### 적용 범위

- **채굴기 (철, 구리, 석탄)**: 지대 제한 적용
- **바이오매스 수집기**: 지대 제한 **미적용** (지표면 수집이므로 제한 없음)
  - 바이오매스는 발전기 연료로 사용되므로 제한 시 전력 확보 자체가 불가능해짐
  - 바이오매스 수집기의 확장 제한은 전력 시스템(FP-002)이 자연스럽게 담당

#### 초기 상태

- 게임 시작 시 철 지대 **1곳** 제공
- 지대당 채굴기 **1대** 배치 → 초반 자원 흐름이 자연스럽게 제한됨

#### 탐색 (Exploration)

| 항목 | 내용 |
|------|------|
| 해금 조건 | 연구 `exploration_basics` 완료 |
| 탐색 비용 | 연료(바이오매스/코크스) + 자원(철판, 구리봉 등) |
| 탐색 시간 | **매우 김** (30분~수 시간, 진행 단계별 증가) |
| 탐색 결과 | 랜덤 자원 지대 발견 (종류, 크기, 품질 결정) |
| 동시 탐색 | 초기 1건, 연구로 최대 3건까지 확장 |

#### 탐색 진행 단계

```
탐색 1회차: 30분  → 철 지대
탐색 2회차: 1시간 → 구리 또는 석탄 지대 (연구 필요)
탐색 3회차: 2시간 → 철/구리/석탄 중 랜덤
탐색 4회차~: 3시간+ → 희귀 자원 지대
```

#### 연구 의존성

| 연구 ID | 이름 | 선행 조건 | 해금 내용 |
|---------|------|----------|----------|
| `exploration_basics` | 기초 탐사 | Phase A 완료 | 탐색 기능 해금, 1슬롯 |
| `advanced_exploration` | 고급 탐사 | 빨간 과학팩x30 | 탐색 슬롯 +1, 구리/석탄 지대 발견 가능 |
| `deep_survey` | 심층 조사 | 파란 과학팩x50 | 탐색 슬롯 +1, 희귀 자원 지대 확률 증가 |

### 기존 설계 변경점

- **DESIGN_DRAFT.md 2-1 생산 기계**: 채굴기 건설 시 빈 슬롯이 있는 자원 지대 선택 필요
- **UI**: 자원 탭 또는 별도 "탐색" 탭에 지대 목록 + 탐색 큐 표시
- **오프라인 진행**: 탐색은 오프라인에서도 진행됨

### 밸런스 의도

- 초반: 채굴기 1대 → 자원 흐름이 느려 전략적 선택 강조
- 중반: 탐색으로 점진적 확장 → "한 번만 더" 심리 유발 (탐색 완료 대기)
- 후반: 다수의 지대 → 대규모 생산 체인 구축 가능

## FP-002: 전력 On/Off 관리 시스템 (Power Grid Management)

### 배경

현재 설계(DESIGN_DRAFT.md 3-2)에서는 전력 부족 시 **비율 기반 감속**이 적용된다:

```
전력비율 = min(1, totalProduction / totalConsumption)
실제생산속도 = 기본생산속도 * 전력비율
```

이 방식은 모든 기계가 균등하게 느려지므로 플레이어의 전략적 선택이 줄어든다.

### 핵심 아이디어

- 전력 부족 시 비율 감속 대신, **시설 On/Off** 방식으로 전력 관리
- 플레이어가 직접 시설을 켜고 끄며 소비 전력을 공급 이하로 맞춤
- 공급 초과 시 **랜덤 강제 Off** 발생 → 즉각적 페널티

### 상세 설계

#### 기계 상태

| 상태 | 설명 |
|------|------|
| `ON` | 정상 가동, 전력 소비 중 |
| `OFF` (수동) | 플레이어가 끈 상태, 전력 소비 0 |
| `OFF` (강제) | 과부하로 시스템이 끈 상태, 전력 소비 0 |

#### 전력 관리 규칙

```
공급전력 = sum(발전기별 발전량)
소비전력 = sum(ON 상태 기계의 전력 소비)

if 소비전력 <= 공급전력:
    모든 ON 기계 정상 가동

if 소비전력 > 공급전력:
    초과분 해소될 때까지 ON 기계 중 랜덤 선택하여 강제 OFF
    강제 OFF된 기계는 "과부하 정지" 상태 표시
```

#### 강제 Off 세부 규칙

| 항목 | 내용 |
|------|------|
| 트리거 | 소비전력 > 공급전력인 순간 즉시 |
| 대상 선택 | ON 상태 기계 중 랜덤 (발전기 제외) |
| Off 수량 | 소비 <= 공급 될 때까지 1대씩 Off |
| 복구 | 플레이어가 수동으로 On (공급 여유 필요) |
| 알림 | "과부하! [기계명] 3대가 강제 정지되었습니다" 토스트 |

#### 우선순위 시스템 (선택적 확장)

플레이어가 기계별 **전력 우선순위**(1~5)를 설정:
- 강제 Off 시 낮은 우선순위부터 Off
- 기본 우선순위: 3 (중간)
- 발전기/연료 공급 기계: 기본 5 (최우선, 변경 불가)

```
우선순위 예시:
5 (최우선): 바이오매스 수집기, 발전기 → 절대 꺼지지 않음
4: 제련기 → 핵심 생산라인
3: 분쇄기, 채굴기 → 일반
2: 조립기 → 여유 시 가동
1: 연구 서버 → 가장 먼저 Off
```

#### UI 변경

- **기계 패널**: 각 기계에 On/Off 토글 버튼 추가
- **전력 바**: `[소비 / 공급]` 형태로 표시, 초과 시 빨간색
- **일괄 제어**: "모두 On", "모두 Off", "우선순위 N 이하 Off" 버튼
- **전력 경고**: 소비가 공급의 90% 이상일 때 경고 표시

```
┌─────────────────────────────────────────┐
│ ⚡ 전력: 185W / 200W  [█████████░] 92%  │  ← 정상 (녹색)
│ ⚡ 전력: 245W / 200W  [██████████] 122% │  ← 과부하 (빨간색, 깜빡임)
└─────────────────────────────────────────┘
```

### 기존 설계 변경점

- **DESIGN_DRAFT.md 3-2 전력 부족 시**: 비율 감속 → On/Off 방식으로 **교체**
- **게임 루프**: 전력비율 계산 대신 On/Off 상태 확인 + 강제 Off 로직
- **오프라인 처리**: 오프라인 중에도 강제 Off 상태 유지, 복귀 시 알림

### 밸런스 의도

- **능동적 관리**: 플레이어가 전력 배분을 직접 결정 → 전략적 깊이 증가
- **긴장감**: 확장 시 전력 한계를 의식해야 함 → 무분별한 기계 증설 억제
- **초반**: 공급 20W, 소비 30W → 일부 기계를 꺼야 하는 선택 강요
- **벌칙의 명확성**: "전체 67% 속도" 보다 "제련기 2대 정지"가 더 직관적

## FP-003: 코어(연산력) 시스템 (Core / Compute System)

### 배경

전력만으로는 시설 확장의 제약이 단일 축에 머문다.
**코어(연산력)**를 두 번째 인프라 자원으로 도입하여, 플레이어가 "전력 + 코어" 두 축을 동시에 관리해야 하는 구조를 만든다.

### 핵심 아이디어

- 시설이 가동될 때 전력 **+ 코어**를 동시에 요구
- 코어를 공급하는 전용 시설(연산 노드)이 존재하며, 이 시설도 전력을 소비
- 전력과 동일하게 **On 상태 시설만** 코어를 소비
- 코어는 저장 불가, 실시간 공급/소비 매칭

### 상세 설계

#### 코어 공급 시설

| 시설 ID | 이름 | 코어 공급 | 전력 소비 | 비용 | 해금 조건 |
|---------|------|----------|----------|------|----------|
| `basic_node` | 기초 연산 노드 | 4 Core | 15W | 철판x20, 회로x5 | 연구 `basic_computing` |
| `advanced_node` | 고급 연산 노드 | 12 Core | 40W | 프레임x15, 회로x20 | 연구 `advanced_computing` |
| `quantum_node` | 양자 연산 노드 | 40 Core | 100W | 프레임x50, 회로x50, 모터x20 | 연구 `quantum_computing` |

#### 기계별 코어 요구량

| 기계 | 전력 소비 | 코어 소비 |
|------|----------|----------|
| 바이오매스 수집기 | 0W | 0 Core |
| 채굴기 (전종) | 5W | 0 Core |
| 분쇄기 | 10W | 1 Core |
| 제련기 | 15W | 2 Core |
| 조립기 | 20W | 3 Core |
| 연구 서버 | 30W | 5 Core |

> 기초 시설(수집기, 채굴기)은 코어 불필요 → 초반 진입장벽 없음
> 가공/연구 시설일수록 코어 요구 증가 → 확장 시 연산 인프라 투자 필요

#### 코어 부족 시 (FP-002와 동일 방식)

```
if 코어소비 > 코어공급:
    코어를 소비하는 ON 기계 중 랜덤 강제 OFF (소비 <= 공급 될 때까지)
    "연산 과부하! [기계명] N대가 정지되었습니다" 토스트
```

- 전력 우선순위 시스템과 통합: 우선순위가 낮은 기계부터 Off
- 전력/코어 중 하나라도 부족하면 강제 Off 발생

#### 전력-코어 이중 제약의 의미

```
예시 상황: 공급 200W / 12 Core

시설 A (제련기 x5): 75W, 10 Core  ← 전력 여유, 코어 빡빡
시설 B (채굴기 x10): 50W, 0 Core  ← 코어 무관
시설 C (연구 서버 x1): 30W, 5 Core ← 코어 부족! 강제 Off 대상

→ 연구 서버를 돌리려면 제련기를 줄이거나 연산 노드를 추가해야 함
→ 연산 노드를 추가하면 전력이 부족해짐
→ 전력을 늘리려면 발전기 연료(바이오매스/코크스)가 더 필요
→ 자원 추출이 제한되어 있으므로 효율적 배분이 핵심
```

### 밸런스 의도

- 전력만으로는 "발전기만 늘리면 됨" → 코어 추가로 **두 번째 병목** 생성
- 연산 노드 자체가 전력을 먹으므로 **자원 경쟁** 심화
- 상위 시설일수록 코어 요구 증가 → 무한 확장 억제

## FP-004: 상위 레시피 시스템 (Advanced Recipe Tiers)

### 배경

현재 설계에서는 하나의 생산품에 하나의 레시피만 존재한다.
연구를 통해 **같은 결과물의 상위 레시피**를 해금하면, 자원 효율은 올라가되 시설/전력/코어 비용이 증가하는 트레이드오프를 만든다.

### 핵심 아이디어

- 동일 결과물에 대해 **Tier 1 → Tier 2 → Tier 3** 레시피 존재
- 상위 레시피는 **변환 비율 개선** 또는 **부재료 추가로 산출량 증가**
- 상위 레시피는 **상위 시설**에서만 사용 가능 (전력/코어 더 소비)

### 상세 설계

#### 레시피 예시: 철판 생산

| Tier | 레시피 | 입력 | 출력 | 사용 시설 | 전력 | 코어 |
|------|--------|------|------|----------|------|------|
| T1 | 기본 제련 | 원석 x2 | 철판 x1 | 제련기 | 15W | 2 |
| T2 | 고효율 제련 | 원석 x3 | 철판 x2 | 고급 제련기 | 25W | 4 |
| T3 | 촉매 제련 | 원석 x3, 촉매 x1 | 철판 x3 | 정밀 제련기 | 40W | 6 |

> T1: 원석 2 → 철판 1 (효율 0.50)
> T2: 원석 3 → 철판 2 (효율 0.67) — 원석당 33% 향상
> T3: 원석 3 + 촉매 1 → 철판 3 (효율 1.00) — 원석당 100% 향상, 단 촉매 필요

#### 레시피 예시: 회로 생산

| Tier | 레시피 | 입력 | 출력 | 사용 시설 |
|------|--------|------|------|----------|
| T1 | 기본 조립 | 철판 x2, 구리봉 x1 | 회로 x1 | 조립기 |
| T2 | 정밀 조립 | 철판 x3, 구리봉 x1 | 회로 x2 | 고급 조립기 |
| T3 | 나노 조립 | 철판 x2, 구리봉 x1, 촉매 x1 | 회로 x3 | 정밀 조립기 |

#### 상위 시설

| 시설 ID | 이름 | 사용 가능 Tier | 전력 | 코어 | 비용 | 해금 조건 |
|---------|------|---------------|------|------|------|----------|
| `smelter` | 제련기 | T1 | 15W | 2 | 철판x15 | 시작 |
| `adv_smelter` | 고급 제련기 | T1~T2 | 25W | 4 | 프레임x20, 모터x5 | 연구 |
| `prec_smelter` | 정밀 제련기 | T1~T3 | 40W | 6 | 프레임x40, 회로x20, 모터x10 | 연구 |
| `assembler` | 조립기 | T1 | 20W | 3 | 프레임x5, 회로x2 | 연구 |
| `adv_assembler` | 고급 조립기 | T1~T2 | 35W | 5 | 프레임x25, 회로x15 | 연구 |
| `prec_assembler` | 정밀 조립기 | T1~T3 | 55W | 8 | 프레임x60, 회로x40, 모터x15 | 연구 |

> 상위 시설은 하위 Tier 레시피도 사용 가능 (하위 호환)
> 상위 시설은 항상 전력/코어를 더 소비 → 효율 개선 vs 인프라 비용 트레이드오프

#### 촉매 (Catalyst)

- 새로운 중간재로 T3 레시피에서 부재료로 사용
- 생산 자체에도 상위 시설과 자원이 필요 → 후반 콘텐츠
- 촉매는 소모성 (투입 시 사라짐)

#### 연구 트리

```
기본 제련 (시작)
  └→ 고효율 제련 연구 → T2 제련 레시피 + 고급 제련기 해금
       └→ 촉매 제련 연구 → T3 제련 레시피 + 정밀 제련기 해금
            └→ 촉매 합성 연구 → 촉매 생산 레시피 해금

기본 조립 (연구)
  └→ 정밀 조립 연구 → T2 조립 레시피 + 고급 조립기 해금
       └→ 나노 조립 연구 → T3 조립 레시피 + 정밀 조립기 해금
```

### 밸런스 의도

- **자원 효율 vs 인프라 비용**: T3가 원석 효율은 최고지만 전력/코어/촉매 부담 큼
- **선형 성장 유도**: 레시피 개선은 2~3배 수준 → 지수적 폭발 방지
- **선택의 깊이**: 제한된 자원 추출(FP-001) 하에서 T1 다수 vs T3 소수 전략 분기

## 밸런스 철학: 브레이킹 구조 종합

위 제안들(FP-001 ~ FP-004)은 하나의 밸런스 철학을 공유한다:

### 제한된 입력, 개선 가능한 처리

```
[자원 추출] ──→ [가공/생산] ──→ [최종 산출물]
   │                │
   │ FP-001:        │ FP-003: 코어 제한
   │ 지대 제한       │ FP-002: 전력 On/Off
   │ (하드캡)        │ FP-004: 상위 레시피 (효율 개선)
   │                │
   ▼                ▼
  느리게 증가      점진적 개선 (선형)
  (탐색 시간)     (연구 → 레시피 → 시설)
```

### 브레이킹 3원칙

| 원칙 | 메커니즘 | 효과 |
|------|---------|------|
| **1. 입력 하드캡** | 자원 지대 수 = 최대 채굴기 수 (지대당 1대) | 아무리 노력해도 원석 유입량에 천장 존재 |
| **2. 효율 개선은 선형** | T1→T2: +33%, T2→T3: +50% (부재료 필요) | 지수 성장이 아닌 점진적 개선 |
| **3. 인프라 비용 증가** | 상위 레시피 → 더 많은 전력/코어 → 더 많은 발전기/노드 → 더 많은 자원 소비 | 효율 개선의 일부를 인프라가 잡아먹음 |

### 유저 경험 흐름

```
1단계: "채굴기가 1대뿐이네, 더 짓고 싶다" → 탐색 시작 (30분 대기)
2단계: "지대 늘었지만 전력이 부족해" → 발전기 건설, 시설 On/Off 관리
3단계: "전력 해결했는데 코어가 부족해" → 연산 노드 건설 (전력 더 필요)
4단계: "생산량 한계에 왔다, 레시피를 개선하자" → T2 연구 (시간 투자)
5단계: "T2로 효율 올랐지만 전력/코어가 또 부족" → 인프라 재조정
6단계: "촉매까지 만들어서 T3 돌리자" → 복잡한 생산 체인 관리

→ 매 단계마다 "한 단계 더" 동기부여, 하지만 생산량은 점진적 증가
→ 지수적 폭발 없이 끊임없는 최적화 욕구 유발
```
